# 《造梦西游》素材整理规范

状态：2026-07-21 起生效  
适用范围：《造梦西游 1》《造梦西游 2》《造梦西游 3》及后续版本的提取、分类、去重、命名、筛选和运行时选材。

## 目标

素材整理需要同时满足三件事：

1. 能从原 SWF 和代码追溯素材来源；
2. 分类目录简洁、可理解，不堆放运行时可生成的冗余项；
3. Godot 项目只使用实现功能所需的标准素材。

## 三层目录职责

### 1. 溯源层

- `sources/raw`
- `sources/decoded`
- `assets/extracted/full/<game>`

这些目录保存原始文件和完整提取结果。默认不做精简删除，确保任何人工判断都可重新验证和再生成。

### 2. 标准分类层

- `assets/extracted/classified/<game>`

这是人工整理后的标准素材库。统一语义目录和名称，删除重复格式、空白图、运行时冗余帧及无保留价值的分类副本。

### 3. 开发选材层

- `assets/selected/<game>`
- `assets/ui`、`resources` 等运行时目录

这里只放实际开发确认采用的素材和配置。选材层可以加工、裁切或规范化，但必须保留来源清单，不能代替溯源层。

## 判断顺序

整理一组素材前，依次确认：

1. 它来自哪个 SWF、符号、脚本或时间轴；
2. 它在原游戏中承担什么功能；
3. 多张图之间属于真实动画、数值变化、语义枚举、组合零件，还是重复格式；
4. Godot 运行时需要逐帧资源，还是可以由控件、文本、Shader 或组合节点生成；
5. 最终保留项是否足以复现原效果。

源码、`symbolClass`、FrameLabel 和实际调用优先于文件名猜测。无法确认的素材进入待确认目录，不强行命名或删除。

## 通用保留与精简规则

### 真实动画

动作、特效、闪光、变色、波纹、粒子或形变等随时间改变画面内容的序列，应保留有效动画帧。删除时只处理完全空白、重复停顿或已验证无引用的帧。

### 数值驱动时间轴

如果 Flash 只是依据生命、魔法、经验、能量、加载进度等数值调用 `gotoAndStop(...)`，而各帧只改变同一图形的可见长度：

- 每种样式只保留一张 100% 完整填充图；
- 底槽、边框、头像和文字底板等必要静态图层分别保留；
- 0%～99% 的中间帧不进入标准分类库；
- Godot 使用 `TextureProgressBar`、裁切区域或 Shader 生成比例；
- 需要复刻原版整数跳变时设置 `step = 1`，或先把比例四舍五入为整数。

完整填充帧不固定是第 1 帧，必须结合代码公式和画面确认。例如：

- `gotoAndStop(round(100 * (1 - current / maximum)) + 1)`：通常第 1 帧是满值；
- `gotoAndStop(round(100 * current / maximum))`：通常末端帧是满值。

### 语义枚举

Boss 名称、角色头像、装备等级、按钮状态等虽然可能位于同一个 MovieClip 中，但每帧含义不同。此类素材按语义命名并保留所有实际使用的变体，不能当作百分比冗余帧删除。

### 组合零件

角色换装、武器、防具和分层特效需要根据原代码的组合方式整理。单张零件不能被误认为完整皮肤；应记录角色、部位、showid、层级、锚点和来源符号。

### 文本、数字与简单图形

普通文本、伤害数字、纯色底板和简单边框优先由 Godot 字体、Theme、StyleBox 或 Shader 生成。只有原作字体造型、纹理或特殊效果无法等价生成时，才保留位图。

### 重复格式与重复文件

- 字节完全相同：保留一个标准路径。
- PNG、SVG、JPG 视觉等价：优先保留透明度和像素还原最可靠的格式，删除分类副本前记录比对证据。
- 同一素材被多个包重复导出：标准分类层只保留一个规范副本，清单记录全部来源。
- 不修改溯源层来实现分类去重。

## 命名规则

- 顶层使用稳定语义：`人物 / 怪物 / UI / 场景与地图 / 公共元件`。
- 动画使用动作名和补零序号：`idle_001.png`、`attack1_001.png`、`hurt_001.png`。
- 数值条使用功能名：`hp_slider.png`、`mp_slider.png`、`energy_slider.png`、`exp_slider.png`；底槽追加 `_bg`。
- 装备和换装保留 `showid`：`showid_04/part_177.png`。
- 未确认内容使用待确认目录和中性编号，不把猜测写入正式名称。
- 文件名表示素材语义，不沿用没有信息量的 `ui_0001`，除非暂时无法可靠识别。

## 删除与记录要求

标准分类层的删除必须满足至少一项：

- 已确认是数值驱动中间帧；
- 已确认是空白或无内容帧；
- 已确认与保留项字节相同或视觉等价；
- 已确认是提取器产生、运行时不会使用的冗余副本；
- 有代码、符号或引用链证据证明不需要保留。

每次整理在 `sources/manifests` 记录：

- 游戏版本与来源 SWF；
- 原符号、脚本或时间轴；
- 原始数量和分类；
- 保留项及其标准路径；
- 删除范围与判断理由；
- Godot 中的替代实现；
- 尚未确认的问题。

## 标准工作流程

1. 只读检查原 SWF、完整拆包、代码和现有清单。
2. 建立候选清单，区分动画、数值、枚举、零件和重复项。
3. 抽查关键帧或组合效果，确认判断。
4. 先确定标准目录、名称和唯一保留项。
5. 更新规则或版本清单。
6. 清理标准分类层，不破坏溯源层。
7. 检查文件数量、哈希、JSON 和引用路径。
8. 在 Godot 中验证关键状态或动画。

## 造 2、造 3 与后续版本

后续版本不照搬造 1 的具体文件名或帧号，但必须照搬本规范的判断流程。每个版本都要重新核对代码公式、时间轴方向、实际画面和运行时需求，再形成自己的版本清单。
